# Motivation and Scope

As a developer coming from other language than C++, i grew acusstomed to using Enums for passing Flag around.([examples](#example-of-flags-from-other-languages))
However such behavior is noticebly absent from C++ language.

We propose allowing `enum class` to be extended with binary operators, enabling it to behave like [BitmaskType](https://en.cppreference.com/cpp/named_req/BitmaskType).
```C++
enum class Permission {
  None    = 0,
  read    = 1 << 0,
  Write   = 1 << 1,
  Execute = 1 << 2,
};

int main () {
    // Combine permissions
    Permission user_perms = Permission::Read | Permission::Write;

    if (user_perms & Permission::READ == Permission::Read)
        std::cout << "Can read";
}
```

This type of use case is a sought-after feature and has led to the development of many standalone solutions.

- The original and first bitmask solution was proposed by Anthony Williams: [Using Enum Classes as Bitfields](https://www.justsoftwaresolutions.co.uk/cplusplus/using-enum-classes-as-bitfields.html). 
- Evolution to Anthonys's proposal with C++23 concepts: [C++20 Concepts applied - Safe bitmasks using scoped enums](https://andreasfertig.com/blog/2024/01/cpp20-concepts-applied/)
- Standalone Header Library for bitmask behavior: [Bitmask operators and typesafe comparisons for enum class](https://github.com/Dalzhim/ArticleEnumClass-v2)
- Bit Flag Values [Enumerations in C++](https://dev.to/pauljlucas/enumerations-in-c-pn9)
- Stackoverflow questions from developers looking for bitmask behavior from `enum class`
  - [C++ enum flags vs bitset](https://stackoverflow.com/questions/45541760/c-enum-flags-vs-bitset)
  - [How to use bitmask_operators.hpp with namespace and classes](https://stackoverflow.com/questions/42481154/how-to-use-bitmask-operators-hpp-with-namespace-and-classes)
  
And it can reduce alot of boiler plate code, as can be seen here:
[perms.h](https://github.com/llvm/llvm-project/blob/main/libcxx/include/__filesystem/perms.h)
[byte.h](https://github.com/llvm/llvm-project/blob/main/libcxx/include/__cstddef/byte.h)
[perm_options.h](https://github.com/llvm/llvm-project/blob/main/libcxx/include/__filesystem/perm_options.h)


## Example of using Enum as Bitmask from mainstream languages

### Python
```python
from enum import IntFlag

class Permission(IntFlag):
    READ    = 1 << 0  # 0b001
    WRITE   = 1 << 1  # 0b010
    EXECUTE = 1 << 2  # 0b100

# Combine permissions
user_perms = Permission.READ | Permission.WRITE

# Check permissions
if user_perms & Permission.READ:
    print("Can read")
```

### C\#
```C#
[Flags]
public enum Permission
{
    None    = 0,
    Read    = 1 << 0, // 0b001
    Write   = 1 << 1, // 0b010
    Execute = 1 << 2, // 0b100
}

class Program
{
    static void Main()
    {
        // Combine permissions
        Permission userPerms = Permission.Read | Permission.Write;

        Console.WriteLine(userPerms);
        // Read, Write

        // Check permissions
        if ((userPerms & Permission.Read) != 0) {
            Console.WriteLine("Can read");
        }
    }
}
```

### Rust
```Rust
// Cargo.toml: bitflags = "2.4" (or latest)
use bitflags::bitflags;

bitflags! {
    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub struct Permissions: u32 {
        const READ    = 0b00000001;
        const WRITE   = 0b00000010;
        const EXECUTE = 0b00000100;
    }
}

fn main() {
    // Combine flags using bitwise OR
    let mut user_perms = Permissions::READ | Permissions::WRITE;

    // Check permissions
    if user_perms.contains(Permission::READ) {
        println!("Can read");
    }
}
```

### Javascript
```Javascript

    EXECUTE: 1 << 2, // 0b100
};

```

## Why not use C style `typedef eum` instead 
Reverting to using C-style enums means losing the benefits of scoped constants and the type safety provided by C++ `enum class`.
       
```C
typedef enum Permission {
    PERMISSION_NONE    = 0,
    PERMISSION_READ    = 1u << 0, // 0b001
    PERMISSION_WRITE   = 1u << 1, // 0b010
    PERMISSION_EXECUTE = 1u << 2, // 0b100
} Permission;

int main(void)
{
    Permission user_perms = PERMISSION_READ | PERMISSION_WRITE;

    // Check permissions
    if (user_perms & PERMISSION_READ) {
        printf("Can read\n");
    }
}
```

## Can i use std::bitset instead?

std::bitest offers bitwise operation on bits, however it is limited in other aspects.
 
### Doesn't offer same ergonomics as class enum
and requires extension by user, for example:
```C++
#include <bit>
#include <bitset>
#include <cassert>
#include <iostream>
#include <stdexcept>

class BitSetA : public std::bitset<8> {
public:
    // Users have to create meaningful function names or additional abstractions to differentiate the individual bit names
    bool is_write_enabled() {
        return test(2);
    }

    void toggle_write_enabled() {
        flip(2);
    }
};

int main () {
    BitSetA a;
    std::cout << a.is_write_enabled() << std::endl;
    a.toggle_write_enabled();
    std::cout << a.is_write_enabled() << std::endl;
}
```

### no type-safety
`std::bitset` also does not provide the same type-safety guarantees as `enum class`. 
The resulting type remains the generic `std::bitset` type, so the original semantic type information is lost during operations.
[C++ enum flags vs bitset](https://stackoverflow.com/questions/45541760/c-enum-flags-vs-bitset)
```C++
#include <bit>
#include <bitset>
#include <cassert>
#include <iostream>
#include <stdexcept>

class BitSetA : public std::bitset<8> { };
class BitSetB : public std::bitset<8> { }

int main () {
    BitSetA a;
    BitSetB b;
    
    // a and b decay into std::bitset and lose type safety
    std::bitset<8> c = a & b;
}
```

### Passing a `Class` around has different implications across platforms
Unlike a `class enum`. In some cases, classes must be passed as pointers to objects and cannot be passed directly through registers.
This has implications on performance.

# Design

The design is building upon the solution proposed by Andreas Fertig [C++20 Concepts applied - Safe bitmasks using scoped enums](https://andreasfertig.com/blog/2024/01/cpp20-concepts-applied/) which in itself is and extension to [Using Enum Classes as Bitfields](https://www.justsoftwaresolutions.co.uk/cplusplus/using-enum-classes-as-bitfields.html) initially devised by Anthony Williams 

## Design considiration
1. No implicit conversion: It should not be posible to assign Enum of different types to each other, or do any kind of operation.  because with they don't decay into integer. I cannot compare different type of enums togather.
2. User should be able to use bitwsie operation normally as they do with plain C style int 
3. Enabling Bitflag behavior is easy and simple for the user, but it has to be opt in by the user.
4. Can be added to existing enum class

## Usage example
Declare new class:
```C++
enum class [[=bitmask_type]] Permission {
  None    = 0,
  read    = 1 << 0,
  Write   = 1 << 1,
  Execute = 1 << 2,
};

int main () {
    // Combine permissions
    Permission user_perms = Permission::Read | Permission::Write;

    if (user_perms & Permission::READ == Permission::Read)
        std::cout << "Can read";
}
```

Extend existing class to function as bitmask_type:
```C++
enum class Permission {
  None    = 0,
  read    = 1 << 0,
  Write   = 1 << 1,
  Execute = 1 << 2,
};

enable_bitmask_type(Permission);

int main () {
    // Combine permissions
    Permission user_perms = Permission::Read | Permission::Write;

    if (user_perms & Permission::READ == Permission::Read)
        std::cout << "Can read";
}
```

## Reference Implementation
1. On godbolt [https://godbolt.org/z/K6PG7qeK1](https://godbolt.org/z/K6PG7qeK1)
2. Trying locally with gcc [here](https://github.com/amfern/wg21-papers/tree/p000000)

```C++
#include <type_traits>
#include <utility>
#include <meta>

inline constexpr struct BitmaskType{} bitmask_type;

template<typename T>
concept BitmaskTypeLike = std::is_enum_v<T> and (std::meta::has_annotation(^^T, bitmask_type) or requires(T e) {
  enable_bitmask_type(e);
});

template<BitmaskTypeLike T>
constexpr auto
operator|(const T lhs, const T rhs)
{
  return static_cast<T>(std::to_underlying(lhs) |
                        std::to_underlying(rhs));
}

template<BitmaskTypeLike T>
constexpr auto
operator&(const T lhs, const T rhs)
{
  return static_cast<T>(std::to_underlying(lhs) &
                        std::to_underlying(rhs));
}

template<BitmaskTypeLike T>
constexpr auto
operator^(const T lhs, const T rhs)
{
  return static_cast<T>(std::to_underlying(lhs) ^
                        std::to_underlying(rhs));
}

template<BitmaskTypeLike T>
constexpr auto
operator~(const T lhs)
{
  return static_cast<T>(~std::to_underlying(lhs));
}

template<BitmaskTypeLike T>
constexpr auto
operator|=(const T &lhs, const T rhs)
{
  lhs = static_cast<T>(std::to_underlying(lhs) |
                        std::to_underlying(rhs));
  return lhs;
}

template<BitmaskTypeLike T>
constexpr auto
operator&=(const T &lhs, const T rhs)
{
  lhs = static_cast<T>(std::to_underlying(lhs) &
                        std::to_underlying(rhs));
  return lhs;
}

template<BitmaskTypeLike T>
constexpr auto
operator^=(T &lhs, const T rhs)
{
  lhs = static_cast<T>(std::to_underlying(lhs) ^
                       std::to_underlying(rhs));
  return lhs;
}
```
