# Proposal

We allow enabling bitmask operators for enum class types, providing behavior similar to [bitmask.types]{.sref}. 
```C++
enum class [[=std::bitmask::bitmask_type]] Permission {
  None    = 0,
  read    = 1 << 0,
  Write   = 1 << 1,
  Execute = 1 << 2,
};

int main () {
    // Combine permissions
    Permission user_perms = Permission::Read | Permission::Write;

    if (user_perms & Permission::READ == Permission::Read)
        std::cout << "Can read";
}
```

# Motivations
## Motivation 1: Drawing inspiration from other programming languages.

As a developer coming from other language than C++, i grew accustomed to using Enums for passing Flag around.
However such behavior is noticeably absent from C++ language.

### Bitmask in: Python
```python
from enum import IntFlag

class Permission(IntFlag):
    READ    = 1 << 0  # 0b001
    WRITE   = 1 << 1  # 0b010
    EXECUTE = 1 << 2  # 0b100

# Combine permissions
user_perms = Permission.READ | Permission.WRITE

# Check permissions
if user_perms & Permission.READ:
    print("Can read")
```

### Bitmask in: C\#
```C#
[Flags]
public enum Permission
{
    None    = 0,
    Read    = 1 << 0, // 0b001
    Write   = 1 << 1, // 0b010
    Execute = 1 << 2, // 0b100
}

class Program
{
    static void Main()
    {
        // Combine permissions
        Permission userPerms = Permission.Read | Permission.Write;

        Console.WriteLine(userPerms);
        // Read, Write

        // Check permissions
        if ((userPerms & Permission.Read) != 0) {
            Console.WriteLine("Can read");
        }
    }
}
```

### Bitmask in: Rust
```Rust
// Cargo.toml: bitflags = "2.4" (or latest)
use bitflags::bitflags;

bitflags! {
    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub struct Permissions: u32 {
        const READ    = 0b00000001;
        const WRITE   = 0b00000010;
        const EXECUTE = 0b00000100;
    }
}

fn main() {
    // Combine flags using bitwise OR
    let mut user_perms = Permissions::READ | Permissions::WRITE;

    // Check permissions
    if user_perms.contains(Permission::READ) {
        println!("Can read");
    }
}
```

# Considered alternatives solutions

## Alternatives: Why not use C style `typedef enum` instead? 
Reverting to using C-style enums means losing the benefits of scoped constants and the type safety provided by C++ `enum class`.
       
```C
typedef enum Permission {
    PERMISSION_NONE    = 0,
    PERMISSION_READ    = 1u << 0, // 0b001
    PERMISSION_WRITE   = 1u << 1, // 0b010
    PERMISSION_EXECUTE = 1u << 2, // 0b100
} Permission;

int main(void)
{
    Permission user_perms = PERMISSION_READ | PERMISSION_WRITE;

    // Check permissions
    if (user_perms & PERMISSION_READ) {
        printf("Can read\n");
    }
}
```

## Alternatives: Can i use std::bitset instead?

std::bitest offers bit-wise operation on bits, however it is limited in other aspects.
 
### std::bitset Doesn't offer same ergonomics as class enum
and requires extension by user, for example:
```C++
#include <bit>
#include <bitset>
#include <cassert>
#include <iostream>
#include <stdexcept>

class BitSetA : public std::bitset<8> {
public:
    // Users have to create meaningful function names or additional abstractions to differentiate the individual bit names
    bool is_write_enabled() {
        return test(2);
    }

    void toggle_write_enabled() {
        flip(2);
    }
};

int main () {
    BitSetA a;
    std::cout << a.is_write_enabled() << std::endl;
    a.toggle_write_enabled();
    std::cout << a.is_write_enabled() << std::endl;
}
```

### std::bitset has no type-safety
`std::bitset` also does not provide the same type-safety guarantees as `enum class`. 
The resulting type remains the generic `std::bitset` type, so the original semantic type information is lost during operations.
[C++ enum flags vs bitset](https://stackoverflow.com/questions/45541760/c-enum-flags-vs-bitset)
```C++
#include <bit>
#include <bitset>
#include <cassert>
#include <iostream>
#include <stdexcept>

class BitSetA : public std::bitset<8> { };
class BitSetB : public std::bitset<8> { }

int main () {
    BitSetA a;
    BitSetB b;
    
    // a and b decay into std::bitset and lose type safety
    std::bitset<8> c = a & b;
}
```

# Design

The design is building upon the solution proposed by Andreas Fertig [C++20 Concepts applied - Safe bitmasks using scoped enums](https://andreasfertig.com/blog/2024/01/cpp20-concepts-applied/) which in itself is and extension to [Using Enum Classes as Bitfields](https://www.justsoftwaresolutions.co.uk/cplusplus/using-enum-classes-as-bitfields.html) initially devised by Anthony Williams 

## Design consideration
1. No implicit conversion: It should not be possible to assign Enum of different types to each other, or do any kind of operation.  because with they don't decay into integer. I cannot compare different type of enums together.
2. User should be able to use bitwsie operation normally as they do with plain C style int 
3. Enabling bitmask behavior is easy and simple for the user, but it has to be opt in by the user.

# Implementation

1. On godbolt clang reflection [https://godbolt.org/z/5585jaf69](https://godbolt.org/z/5585jaf69)
1. On godbolt gcc 16.1 [https://godbolt.org/z/Ybxz6o41e](https://godbolt.org/z/Ybxz6o41e)
1. Trying locally with gcc 16.1 [here](https://github.com/amfern/wg21-papers/tree/p000000)


```cpp
#include <type_traits>
#include <utility>
#include <meta>

namespace std {

inline constexpr struct BitmaskType {} bitmask_type;

template <typename T>
concept bitmask-like =
    std::is_enum_v<T> and (std::meta::has_annotation(^^T, bitmask_type));

template <bitmask-like T>
constexpr auto operator|(const T lhs, const T rhs) {
  return static_cast<T>(std::to_underlying(lhs) | std::to_underlying(rhs));
}

template <bitmask-like T>
constexpr auto operator&(const T lhs, const T rhs) {
  return static_cast<T>(std::to_underlying(lhs) & std::to_underlying(rhs));
}

template <bitmask-like T>
constexpr auto operator^(const T lhs, const T rhs) {
  return static_cast<T>(std::to_underlying(lhs) ^ std::to_underlying(rhs));
}

template <bitmask-like T> constexpr auto operator~(const T lhs) {
  return static_cast<T>(~std::to_underlying(lhs));
}

template <bitmask-like T>
constexpr auto operator|=(const T &lhs, const T rhs) {
  lhs = static_cast<T>(std::to_underlying(lhs) | std::to_underlying(rhs));
  return lhs;
}

template <bitmask-like T>
constexpr auto operator&=(const T &lhs, const T rhs) {
  lhs = static_cast<T>(std::to_underlying(lhs) & std::to_underlying(rhs));
  return lhs;
}

template <bitmask-like T> constexpr auto operator^=(T &lhs, const T rhs) {
  lhs = static_cast<T>(std::to_underlying(lhs) ^ std::to_underlying(rhs));
  return lhs;
}

}
```
